# Infrastructure Pattern
## Двухрепозиторная архитектура как паттерн для любого курса

> **Course-agnostic документ.** Описывает инфраструктурный паттерн, поддерживающий методологию. Работает для любого предметного курса.

---

# 1. Ключевое разделение: мышление vs действие

В СМД-подходе мыследеятельность имеет три слоя — мыследействие, мысль-коммуникация, чистое мышление. Инфраструктура должна **поддерживать разделение** этих слоёв.

Когда мышление и действие живут в одном инструменте — они смешиваются и выхолащиваются. Когда они разделены — каждый сохраняет свою дисциплину.

Поэтому — **два репозитория** и **два Claude-инструмента**.

---

# 2. Два репозитория

## Репо 1. `smd-learning` — методологический корпус

**Что там:**
- Методологическое ядро (`core/`) — course-agnostic
- Инстанциации курсов (`courses/[курс]/`) — course-specific
- Личный корпус рефлексий и evidence

**Bounded Context:** мышление и обучение

**Статус:** **приватный**. Это личная интеллектуальная собственность. Не идёт в public portfolio.

**Жизненный цикл:** растёт через годы и разные курсы. Не удаляется, не архивируется.

## Репо 2. `[project]` — проектный код

**Что там:**
- Сквозной проект курса (например, Java backend, Rust CLI, архитектурные artifacts)
- Ассессментные упражнения (на старте)
- Реальный production-like код

**Bounded Context:** код и система (в предметной области курса)

**Статус:** приватный сначала, потенциально public позже как portfolio.

**Жизненный цикл:** завершается вместе с курсом. Может переходить в новый курс или в реальную работу.

## Почему два репо, а не один

**FPF-обоснование:**
- Разные Bounded Context — разная семантика изменений, разная privacy, разный жизненный цикл
- Single Source of Truth — каждый репозиторий один источник для своей области
- Auditability — историю учёбы не размывает историей кода

**СМД-обоснование:**
- Мышление не должно смешиваться с действием
- Рефлексия требует отдельного "места", отдельной дисциплины
- Переход между репо — осознанный Speech Act переключения деятельности

**Практическое обоснование:**
- Личные рефлексии остаются личными, даже когда код идёт в public
- Scaling — второй курс добавляется в `smd-learning/courses/`, не трогая первый репо
- Claude Code работает чище, когда в каждом репо — свой CLAUDE.md с узким контекстом

# 3. Два Claude-инструмента

## Инструмент 1. Claude Project (chat) — пространство мышления

**Что делает:**
- Ведёт учебные сессии (СМД-цикл)
- Играет 5 позиций
- Проблематизирует
- Проводит мета-сессии
- Строит схемы через artifact-визуализации

**Что не делает:**
- Не пишет в файлы (только предлагает формулировки)
- Не коммитит в git
- Не запускает код

**Конфигурация:**
- Custom Instructions — системный промпт курса
- Files — подключение к `smd-learning` через GitHub connector (предпочтительно) или ручная загрузка

## Инструмент 2. Claude Code — пространство действия

**Что делает:**
- Работает с реальным кодом в проектном репо
- Запускает команды, анализирует вывод
- Применяет изменения в файлах (код и артефакты обучения)
- Коммитит изменения в git

**Что не делает:**
- Не объясняет онтологию предмета (это работа Project)
- Не играет позиции (работает как senior-коллега в парном программировании)
- Не проводит рефлексивные сессии

**Конфигурация:**
- CLAUDE.md в каждом репозитории — контекстный документ
- Работает локально в IDE/terminal

## Асимметрия прав (FPF Commitment)

**Claude Project — READ-ONLY** по отношению к файлам.
**Claude Code — READ + WRITE** по отношению к файлам.

Это принципиальное разделение. Оно обеспечивает:
- Дисциплину осознанных точек фиксации
- Честную синхронизацию (когда учащийся коммитит — это Speech Act)
- Защиту от автоматических изменений без рефлексии

# 4. Структура методологического репозитория

```
smd-learning/
├── README.md                              ← навигация по корпусу
├── CLAUDE.md                              ← контекст для Claude Code в этом репо
│
├── core/                                  ← МЕТОДОЛОГИЧЕСКОЕ ЯДРО (course-agnostic)
│   ├── theoretical-foundations.md         ← СМД + FPF, рабочий дистиллят
│   ├── methodology-method.md              ← сам Method — принципы
│   ├── session-patterns.md                ← паттерны сессий
│   ├── artifact-formats.md                ← форматы артефактов
│   ├── correction-protocol.md             ← 5 контуров + 4 уровня
│   ├── infrastructure-pattern.md          ← этот документ
│   └── course-template.md                 ← шаблон развёртывания нового курса
│
├── methodology-journal.md                 ← КОРНЕВОЙ journal эволюции ядра
│
└── courses/                               ← ИНСТАНЦИАЦИИ
    ├── java-spring-2026/                  ← конкретный курс 1
    │   ├── course-context.md              ← Bounded Context этого курса
    │   ├── course-mechanics.md            ← конкретизация ядра
    │   ├── project-systemprompt.md        ← системный промпт "Java Spring Boot Mastery"
    │   ├── positions-java-world.md        ← 5 Java-специфичных позиций Оппонента
    │   ├── active-module.md               ← текущий активный модуль
    │   ├── modules/                       ← все модули курса
    │   │   └── module-0-audit.md
    │   ├── reflection-log.md              ← логи сессий этого курса
    │   ├── evidence-ledger.md
    │   ├── heat-map.md                    ← появится после Модуля 0
    │   ├── onto-map.md                    ← появится позже
    │   ├── meta-sessions/                 ← мета-сессии курса (контуры 3, 4, 5)
    │   │   └── .gitkeep
    │   └── schemes/
    │       └── .gitkeep
    │
    └── [следующий-курс]/                  ← когда будет второй курс
        └── ...
```

## Правила структуры

- `core/` **не трогается** при обычной работе. Правки — только через методологические диалоги с эмпирической основой.
- `courses/[курс]/` — живой, постоянно изменяется.
- `methodology-journal.md` на **корневом уровне**, не внутри курса. Отражает эволюцию ядра, не курса.
- Meta-sessions курса — внутри курса (`courses/[курс]/meta-sessions/`). Они про траекторию этого курса.

# 5. Структура проектного репозитория

Структура специфична курсу. Но есть общие требования:

- **README.md** в корне — описание проекта, стек, базовые команды
- **CLAUDE.md** в корне — контекст для Claude Code: описание проекта, стандарты кода, режимы работы (парное программирование, позиционный разбор, экзамен)

Остальное — по предмету.

# 6. Два Claude Project

## Проект A: "[Название курса]" — учебный

**Инструкции:** системный промпт курса из `courses/[курс]/project-systemprompt.md`

**Files:**
- `course-mechanics.md` (курсовой)
- `active-module.md`
- `reflection-log.md`
- `evidence-ledger.md`
- `heat-map.md` (когда будет)
- `positions-[predmet].md` (курсовой)

Опционально, из ядра для теоретической опоры:
- `core/theoretical-foundations.md`

**Назначение:** проведение учебных сессий.

## Проект B: "Methodology Development"

**Инструкции:** методологический системный промпт (course-agnostic)

**Files:**
- `core/theoretical-foundations.md`
- `core/methodology-method.md`
- `core/session-patterns.md`
- `core/artifact-formats.md`
- `core/correction-protocol.md`
- `core/infrastructure-pattern.md`
- `methodology-journal.md`

**Назначение:** эволюция Method (уровень D) через методологические диалоги.

**Принципиально:** этот проект **не привязан к конкретному курсу**. Он работает с методологией как таковой. Конкретные курсы упоминаются только как примеры или эмпирическая основа.

# 7. Поток информации между компонентами

```
                    ┌─────────────────────────────────────────┐
                    │   GitHub (источник правды)              │
                    │  ┌──────────────┐  ┌──────────────┐    │
                    │  │ smd-learning │  │   project    │    │
                    │  └──────┬───────┘  └──────┬───────┘    │
                    └─────────┼──────────────────┼───────────┘
                              │                  │
                    READ ▲    │                  │  READ/WRITE ▲
                         │    │                  │             │
                    ┌────▼────┴──┐          ┌────┴─────────────┴──┐
                    │  Claude    │          │   Claude Code       │
                    │  Project   │          │   (локально)        │
                    │            │          │                     │
                    │ - чат      │          │ - реальный код      │
                    │ - позиции  │          │ - коммиты           │
                    │ - рефлексия│          │ - записи в log      │
                    └────────────┘          └─────────────────────┘
```

**Ритм:**
- Учащийся работает в Claude Project — проводит сессию
- Claude Project предлагает формулировки (записи, патчи)
- Учащийся переключается в Claude Code — применяет, коммитит, пушит
- Claude Project читает обновления через GitHub connector (если доступен) или через ручную синхронизацию

# 8. Коммит-конвенция

Для методологического репозитория:

**Артефакты курса:**
- `session #N: [тема]` — запись в reflection-log
- `evidence: claim #ID [status]` — изменение в ledger
- `meta-session #NN: [инсайт]` — мета-запись
- `switch active module: MX -> MY` — переключение модуля
- `scheme: [name] vN` — новая/обновлённая схема

**Правки методологии:**
- `methodology: [суть]` — уровень D в `core/*`
- `course-mechanics: [суть]` — уровень B в курсовом
- `systemprompt: [суть]` — уровень A в курсовом
- `modules: [суть]` — уровень C

**Инфраструктура:**
- `setup: [что]` — создание структуры
- `docs: [что]` — правки README, CLAUDE.md и т.п.

Для проектного репозитория — обычная конвенция разработки плюс:
- `[MX-SN]` префикс в коммитах учебных задач, где M = модуль, S = сессия

# 9. Адаптация при ограничениях инфраструктуры

Не все компоненты могут быть доступны. Нормальные адаптации:

## GitHub connector недоступен

Возможные причины: план Claude.ai не включает, организационные ограничения, технические проблемы.

**Адаптация:** ручная загрузка файлов в Files Claude Project через upload. Раз в 1-2 недели или после значимых коммитов — перезаливать актуальные версии.

**Потери:** больше ручного труда, но функционально эквивалентно.

## Claude Code недоступен

Возможные причины: не установлен, нет доступа.

**Адаптация:** писать в файлы вручную через IDE. Copy-paste формулировки из Claude Project.

**Потери:** медленнее, нет автоматизации bytecode-инспекции и подобного.

## Один Claude вместо двух проектов

**Не рекомендуется**, но возможно в крайнем случае: использовать один Project, в нём и учебные сессии, и методологические диалоги.

**Потери:** смешение контекстов. Риск тихого размывания методологии. Если так — особо жёсткая дисциплина Speech Acts переключения режимов.

---

# 10. Признаки работающей инфраструктуры

- Каждая сессия имеет коммит в `smd-learning`
- Каждая учебная задача имеет коммит в проектном репо
- Claude Project видит актуальный `active-module.md` при запросе
- Mета-сессии имеют записи в `meta-sessions/`
- Methodology-journal растёт медленно, но стабильно (1-3 записи в квартал)
- Ни в одном репозитории нет файлов, которые "забыты" и никто в них не пишет

Если что-то из этого нарушено — инфраструктура дрейфует. Следующая мета-сессия должна это заметить.
