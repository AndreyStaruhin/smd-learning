# Course Template
## Шаблон развёртывания методологии под новый курс

> **Это практический документ.** Описывает, как взять методологическое ядро и инстанциировать его под конкретный предмет — Java, Rust, архитектуру систем, DDD, любой курс.
>
> Работает как чеклист + конструктор.

---

# 1. Когда применять этот шаблон

Используйте этот шаблон, когда:

- Начинаете новый учебный курс по любому предмету
- Переводите существующую учебную программу на эту методологию
- Проектируете методологию для коллеги или команды (с адаптацией)

Не используйте, когда:
- Нужен простой разовый туториал (методология — overkill)
- Предмет не требует онтологического сдвига (просто освоение API, изучение инструмента)
- Учащийся не активный практик (методология требует Claim Scope — профессионала)

# 2. Предварительный аудит (перед началом инстанциации)

## Шаг 1. Определить Bounded Context курса

Ответить на вопросы:

**1.1. Кто учащийся?**
- Какой профессиональный уровень?
- Какие смежные области уже освоены?
- Какая пауза в работе с предметом?
- Какой профессиональный контекст?

Результат: 3-4 предложения точного описания профиля.

**1.2. Какая цель курса?**
- Что учащийся хочет уметь через N месяцев?
- Какой "образ результата"?

Результат: фраза вида "через N месяцев учащийся, получив задачу X, автоматически видит её как Y".

**1.3. Что НЕ включено в курс?**
- Какие смежные темы не входят?
- Какие уровни сложности не достигаются?

Результат: явное "not in scope" — критично для FPF Claim Scope.

**1.4. Какие болевые точки?**
- Какие темы в предмете учащийся избегал?
- Какие парадоксы, противоречия, непонимания накопились?

Результат: список 3-5 точек, которые курс будет специально разбирать.

## Шаг 2. Определить эмпирическую готовность

Проверить:

- Готов ли учащийся к проблематизации собственных решений (не объяснить, а защитить)
- Готов ли к режиму "не знаю" как валидному ответу
- Готов ли к регулярной рефлексии (минимум 5-10 минут после каждой сессии)
- Готов ли к пропуску некомфортных тем **специально**, а не избеганию

Если на любой из вопросов "нет" — методология не сработает. Стоит либо поговорить о готовности, либо использовать более лёгкий формат.

# 3. Артефакты к созданию

## Этап 1. Методологическая подготовка (1 день)

Создать в методологическом репозитории `smd-learning/courses/[имя-курса]/`:

### `course-context.md`

Фиксирует Bounded Context курса. Структура:

```markdown
# [Название курса] — Context

## Bounded Context
[Точное описание профиля учащегося]

## Цель (Claim)
[Что методология обещает]

## Claim Scope
[Для кого работает — явно]

## Образ результата
[Как выглядит учащийся в конце курса]

## Not in scope
[Что курс не покрывает]

## Болевые точки пользователя
[Список известных онтологических пробелов]

## Ожидаемая длительность
[Оценка месяцев/недель при заявленной интенсивности]
```

### `positions-[predmet].md`

Конкретизирует [Оппонент] и [Симулянт-эксперт] под предмет. Структура:

```markdown
# Позиционная карта — [предметная область]

## [Оппонент]-воплощения (онтологические позиции в предмете)

### Позиция 1: "[название]"
**Онтология:** [что существует для этой позиции]
**Ключевые средства:** [какими понятиями мыслит]
**Проблематики:** [что считает проблемой]
**Типичная критика:** [как атакует другие позиции]

### Позиция 2: ...

## [Симулянт-эксперт]-воплощения (индустриальные роли)

### Роль 1: [например, Tech Lead]
**Контекст работы:** [...]
**Что оценивает:** [...]
**Типичный стиль:** [...]

### Роль 2: ...
```

**Минимум:** 3-5 Оппонент-позиций, 3-4 Симулянт-роли. Меньше — бедно для многопозиционности.

### `course-mechanics.md`

Конкретизация ядерных правил под курс. **Не дублирует** `core/session-patterns.md`, а указывает специфику:

- Длительность сессии (может быть 60-120 мин вместо 90)
- Ритм недели
- Специфические триггеры для Claude
- Инструменты мыследействия (IDE, среда, репо)
- Особенности Evidence-уровней для этого предмета

### `project-systemprompt.md`

Системный промпт для учебного Claude Project. Основан на `core/methodology-method.md` + позиции из `positions-[predmet].md`. 

**Обязательно включает:**
- Миссию курса
- Bounded Context (короткая версия)
- 5 позиций с воплощениями
- 5 режимов СМД-цикла
- Триггерные слова
- Правила маркировки позиций
- Асимметрию прав (Project читает, Code пишет)
- Явную ссылку на `active-module.md`

**Не включает:** глубокое теоретическое описание СМД и FPF (для этого есть `core/theoretical-foundations.md`).

### `modules/module-0-[ассессмент].md`

Ассессментный модуль из 5 сессий + финальной сборки heat-map. Структура сессий универсальна, конкретные задачи — под предмет.

**Принципы проектирования ассессмента:**
- Задачи покрывают основные темы курса
- Три уровня сложности (R, T, C) для каждой темы
- Задачи в реальных условиях, не абстрактные
- Критерии оценки сформулированы заранее
- Для каждой задачи — ожидаемый онтологический пробел, который она проявит

### `active-module.md`

На старте — **копия** `modules/module-0-[ассессмент].md`. Меняется при переходах.

### Пустые шаблоны

- `reflection-log.md` — с заголовком и первой тестовой записью #0 про подготовку инфраструктуры
- `evidence-ledger.md` — с пустой таблицей и заголовками колонок
- `meta-sessions/.gitkeep`
- `schemes/.gitkeep`

## Этап 2. Проектная инфраструктура (пол-дня)

Создать отдельный проектный репозиторий. Минимально:

### `README.md`

Описание проекта, связь с учебным курсом, стек, команды запуска.

### `CLAUDE.md`

Контекст для Claude Code. Структура из `core/infrastructure-pattern.md §5`. 

**Обязательно включает:**
- Описание проекта и его связь с методологическим репо
- Технологический стек
- Стандарты кода
- Режимы работы (парное программирование, позиционный разбор, экзамен)
- Часто используемые команды

## Этап 3. Настройка Claude Projects (полчаса)

Создать два Claude Projects:

### Project 1: "[Название курса]"

- Instructions — `project-systemprompt.md`
- Files — методологический репо через GitHub connector (или ручная загрузка)

### Project 2: "Methodology Development"

- Если уже существует (для предыдущих курсов) — **не создавать заново**, подключить актуальный методологический корпус
- Если первый курс — создать по шаблону из `infrastructure-pattern.md §6`

# 4. Специфические решения для предметной области

При инстанциации нужно принять решения по нескольким пунктам:

## 4.1. Онто-карта предмета (первичный набросок)

До начала обучения сделать **набросок** ключевых уровней существования в предмете. Это не замена `onto-map.md` учащегося, а проектный набросок для проектировщика курса.

Для Java Spring: IoC-контейнер → Bean → Configuration → Autoconfiguration → Aspects
Для Rust: Ownership → Borrowing → Lifetimes → Traits → Async
Для DDD: Entity → Value Object → Aggregate → Bounded Context → Context Map
Для архитектуры систем: Request → Service → Data Flow → Consistency → Evolvability

Набросок нужен для того, чтобы **правильно спроектировать модули** — каждый модуль раскрывает один уровень.

## 4.2. Программа модулей

Стартовая версия программы, которая будет **переприоритизирована** по heat-map после Модуля 0.

**Принципы:**
- Каждый модуль имеет явную онтологическую цель ("после модуля учащийся видит X как Y")
- Модули composable — могут перестраиваться по результатам heat-map
- Специальные болевые модули вставляются в приоритет (например, модуль G для generics в Java)
- Каждый модуль определяет свои критерии Evidence (R/T/C)

## 4.3. Сквозной проект

Выбрать один **сквозной проект**, через который будут проходить задачи всех модулей. 

**Критерии хорошего сквозного проекта:**
- Достаточно богат, чтобы все модули имели в нём задачи
- Реалистичен, похож на production-задачу
- Заведомо не решается очевидным применением известных паттернов
- Ценен для учащегося сам по себе (не "учебная" задача)

**Примеры:**
- Для Java Spring — backend-сервис для X
- Для Rust — CLI-инструмент или системный компонент
- Для архитектуры систем — проектирование конкретной сложной системы
- Для DDD — моделирование конкретного домена с коллегой

## 4.4. Инструменты мыследействия

Явно определить:
- В какой IDE работает учащийся
- Какая среда выполнения (container, VM, cloud)
- Какие вспомогательные инструменты (bytecode inspector, profiler, debugger)
- Какой репозиторий для проекта

Всё это записать в `CLAUDE.md` проектного репо.

# 5. Первые две недели курса (on-ramp)

## Неделя 1: Модуль 0 (ассессмент)

- День 1: инфраструктурная готовность
- Дни 2-6: 5 сессий ассессмента
- День 7: финальная сборка heat-map + первая мета-сессия

Детали — в `courses/[курс]/modules/module-0-[ассессмент].md`.

## Неделя 2: стартовый модуль после Модуля 0

По результатам heat-map выбирается следующий модуль. Обычно это тот, где больше всего 🔴/🟠 или тот, который разблокирует следующие.

Первый модуль после ассессмента — **критически важный период**. Здесь проверяется, работает ли методология для этого учащегося, этого предмета, этой инфраструктуры. Поэтому:

- Чаще короткие рефлексии (после каждой сессии + в конце каждого дня)
- Первая мета-сессия курса — на 2-й неделе, не на 4-й
- Готовность быстро менять уровень A (системный промпт) при явных сбоях

# 6. Эволюция курса и методологии

## В ходе курса

- Правки уровня A/B/C в мета-сессиях курса → меняют `courses/[курс]/*`
- Правки уровня D в методологических диалогах → меняют `core/*`
- Мета-сессии курса — раз в 2 недели, не пропускаются
- Методологические диалоги — по необходимости, когда вскрывается уровень D

## По окончании курса

При закрытии последнего модуля и финальной курсовой рефлексии:

1. **Аудит эмпирики:** были ли накоплены данные для правок уровня D? Если да — они фиксируются в `methodology-journal.md`.

2. **Консолидация паттернов:** если за курс серия правок A сложилась в общий паттерн — выносится в `core/*`.

3. **Архивация курса:** `courses/[курс]/` остаётся как есть, не переписывается. Это история и опыт.

4. **Подготовка к следующему курсу:** используется обновлённое ядро (с учётом правок уровня D) + этот шаблон для нового курса.

# 7. Анти-паттерны инстанциации

Ошибки, которые стоит избежать при создании нового курса:

## Анти-паттерн 1. Копирование из предыдущего курса

Создавать новый курс через copy-paste предыдущего и замену названий. Это **не инстанциация**, это дублирование.

**Правильно:** взять `core/` за основу, `courses/[предыдущий]/*` — за **пример** (не шаблон), создать свои документы.

## Анти-паттерн 2. Размытый Bounded Context

"Для всех, кто хочет изучить X". Это Claim без Scope. Методология работает только при узком, точном профиле учащегося.

**Правильно:** конкретный профиль с профессиональным уровнем, контекстом, паузой, болями.

## Анти-паттерн 3. Слишком много позиций

"Давайте добавим 10 позиций Оппонента, чтобы было богаче". Учащийся не сможет всерьёз освоить 10 позиций.

**Правильно:** 3-5 Оппонент-воплощений + 3-4 Симулянт-роли. Больше — не углубляет, а рассеивает.

## Анти-паттерн 4. Ассессмент как самоутверждение

"В ассессменте будут сложные задачи, чтобы учащийся понял, как много не знает". Это не ассессмент, это унижение.

**Правильно:** задачи нарастающей сложности, честно покрывающие предметную область. Цель — измерить, не проверить "достойность".

## Анти-паттерн 5. Спроектировано всё заранее

"Мы спроектировали все 11 модулей в деталях до начала курса". Методология требует гибкости — детальное проектирование модулей **после Модуля 0**, по heat-map.

**Правильно:** до курса — стартовая версия программы (список + краткие описания). Детали каждого модуля — по мере приближения.

## Анти-паттерн 6. Игнорирование эмпирики первого курса

Начинаем второй курс, не учтя ничего из опыта первого. Это — потеря накопленного.

**Правильно:** перед вторым курсом — курсовая рефлексия первого с фокусом на то, что консолидируется в ядро.

# 8. Чеклист развёртывания нового курса

Перед стартом убедиться:

**Методологический репозиторий:**
- [ ] `courses/[имя]/course-context.md` написан
- [ ] `courses/[имя]/positions-[predmet].md` с 3-5 Оппонент-позициями + 3-4 Симулянтами
- [ ] `courses/[имя]/course-mechanics.md` с курсовой спецификой
- [ ] `courses/[имя]/project-systemprompt.md`
- [ ] `courses/[имя]/modules/module-0-[ассессмент].md` с 5 сессиями ассессмента
- [ ] `courses/[имя]/active-module.md` = копия module-0
- [ ] Пустой `reflection-log.md` с записью #0
- [ ] Пустой `evidence-ledger.md` с заголовками
- [ ] Пустые `meta-sessions/` и `schemes/` с .gitkeep

**Проектный репозиторий:**
- [ ] Создан, привязан к GitHub
- [ ] `README.md` написан
- [ ] `CLAUDE.md` с режимами работы

**Claude Projects:**
- [ ] Учебный Project создан, Instructions заполнены
- [ ] Files подключены (connector или upload)
- [ ] Methodology Development Project существует (общий для всех курсов)

**Инфраструктура:**
- [ ] Инструменты установлены (IDE, Claude Code)
- [ ] Проверена работоспособность (простой тест: Claude Project видит active-module.md?)

**Готовность учащегося:**
- [ ] Понимает Claim Scope (для кого методология)
- [ ] Готов к 5-10 часам в неделю в течение [N] месяцев
- [ ] Принимает дисциплину Evidence и регулярной рефлексии

Только после выполнения всех пунктов — приступать к Модулю 0.

# 9. Итог: чем этот шаблон полезен

- **Для вас сейчас:** структурированно развернуть первый курс (Java Spring Boot 3)
- **Для вас через полгода:** быстро развернуть второй курс, переиспользовав всё ядро
- **Для сообщества (если методология станет публичной):** чужой человек может взять ядро + шаблон и адаптировать под свой предмет

Это реализация принципа "шаблон для создания шаблонов" — изначальной цели методологии.
