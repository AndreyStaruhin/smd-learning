# Artifact Formats
## Форматы артефактов методологии, применимые к любому курсу

> **Course-agnostic документ.** Задаёт структуру артефактов, которые ведутся в любом курсе по этой методологии. Конкретное содержимое — инстанциация.

---

# 1. Принципы дизайна артефактов

Каждый артефакт проектируется с учётом FPF-критериев:

**Single Source of Truth:** каждая область информации имеет один авторитетный источник. Не дублируется.

**Auditability:** каждая запись имеет контекст, автора (неявно — учащегося), дату, и при необходимости — ссылку на основание.

**Evolvability:** формат может эволюционировать, но версии фиксируются в git. Старые записи остаются в старом формате (не переписываются).

**Machine-readability:** структура достаточно жёсткая, чтобы Claude Project мог эффективно работать с артефактом — находить записи по темам, сопоставлять даты, отслеживать паттерны.

# 2. Reflection Log

## Назначение

Хронологический лог всех учебных сессий. Основной инструмент рефлексии и источник данных для мета-сессий.

## Правила ведения

- Одна запись на каждую сессию. Без исключений.
- Новые записи — в начало файла (сверху). Старые уходят вниз.
- Запись делается **в конце сессии**, не позже. Запись через день — ложная реконструкция.
- Запись делается через Claude Code (или вручную в IDE), не через Claude Project. Claude Project только предлагает формулировку.

## Формат записи

```markdown
## Сессия #[N] | [YYYY-MM-DD] | Модуль [X] | Тема: [...]

### Контекст
[1-2 предложения: откуда пришли, что предшествовало]

### Позиции в сессии
- Вход: [какая позиция Claude на входе]
- Основная работа: [какие позиции, сколько времени примерно]
- Закрытие: [Методолог обычно]

### Сдвиг содержания
[Что новое теперь понимаю/вижу. НЕ список тем, а ФОРМУЛИРОВКА сдвига.
Плохо: "разобрали generics, type erasure, bridge methods"
Хорошо: "теперь вижу, что type erasure — не костыль, а осознанное решение 
с известными обходными паттернами; раньше воспринимал как ограничение"]

### Сдвиг процесса
[Что заметил о собственном обучении. 1-2 предложения о процессе, не о содержании.
Плохо: "всё понял"
Хорошо: "заметил, что автоматически ищу 'правильный ответ', а не действую; 
нужно сознательно удерживать позицию 'я не знаю — я действую и смотрю'"]

### Открытые вопросы
- [ ] Вопрос 1
- [ ] Вопрос 2

### Evidence-претензия
[Что готов предъявить как свидетельство: коммит, тест, файл схемы, 
PR; либо "не готов предъявить — требует дополнительной работы"]

### Артефакты
- code: [ссылка на коммит в проектном репо]
- scheme: [если построена схема — относительный путь]
- artifact: [если был визуальный артефакт в чате]
```

## Строгие требования к секциям

- "Сдвиг содержания" — **формулировка**, не перечисление тем. Если нет формулировки — написать "сдвиг не зафиксирован, тема требует возврата".
- "Сдвиг процесса" — обязательно про себя, не про предмет.
- "Evidence-претензия" — конкретная ссылка либо честное "не готов".
- Пустых секций быть не должно. Пустота = "н/а с обоснованием".

## Анти-паттерн

Записи начинают звучать одинаково, "сдвиг" превращается в штамп. Это сигнал либо плато в обучении, либо ритуализации рефлексии — разбирается на ближайшей мета-сессии как основная тема.

# 3. Evidence Ledger

## Назначение

Реестр всех проверяемых claims освоения тем. Основа для дисциплины Evidence.

## Правила ведения

- Запись заводится при **формулировании claim** (не при закрытии)
- Обновление через Claude Code
- Каждый claim имеет ID, статус, уровень, ссылку на evidence
- **Falsified** claims не удаляются, а помечаются — это часть честной истории

## Формат

```markdown
# Evidence Ledger

## Реестр

| ID | Модуль | Claim | Level | Evidence | Дата | Статус |
|----|--------|-------|-------|----------|------|--------|
| 001 | M0 | Владею современными фичами Java 21 | R | A1.1, A1.2, A1.3 | 2026-04-20 | ✅ closed |
| 042 | M5 | Могу диагностировать N+1 и применить @EntityGraph | C | exam M5, session #28 | TBD | 🟡 in-progress |
| 017 | M2 | Понимаю bean lifecycle на уровне онтологии | T | session #9 scheme, PR#4 | 2026-05-03 | ❌ falsified |

## Метрики (обновляются после каждой мета-сессии)

- **Всего claims:** N
- **Closed:** N (XX%)
- **In-progress:** N
- **Falsified:** N
- **По уровням:** R=N, T=N, C=N
```

## Определения уровней

- **R (Reproductive):** знакомый паттерн в знакомой ситуации
- **T (Transfer):** знакомый паттерн в новой ситуации узнаваемого типа
- **C (Constructive):** новое решение в ситуации, где известные паттерны не работают

## Определения статусов

- **⬜ planned:** claim сформулирован, evidence не собрано
- **🟡 in-progress:** работа идёт, evidence частично есть
- **✅ closed:** evidence собрано и проверено (обычно через экзамен или ревизию)
- **❌ falsified:** claim был закрыт, но последующая проверка выявила, что освоение поверхностно. **Важная категория** — её наличие признак честной методологии, отсутствие — признак ложной "успешности".

## Дисциплина

- Claim не закрывается автоматически по факту прохождения упражнений. Закрытие — Speech Act через экзамен или мета-сессию.
- Claim уровня C не закрывается без задачи, где известные паттерны не работали.
- Статус ❌ не повод для самокритики, это результат работающей falsifiability.

# 4. Heat-Map

## Назначение

Индивидуальная карта онтологических пробелов учащегося. **Определяет фактическую программу курса.**

## Когда строится

- Впервые — в конце Модуля 0 (ассессмент)
- Пересматривается — при переходе между крупными блоками модулей или когда в мета-сессии вскрывается смещение реальности vs плана

## Формат

```markdown
# Heat Map — [YYYY-MM-DD]

## Диагностика по темам

| Тема | Уровень | Приоритет | Комментарий |
|------|---------|-----------|-------------|
| [Тема 1] | 🟢 | Low | [короткий комментарий] |
| [Тема 2] | 🟠 | P1 | [что нужно проработать] |
| [Тема 3] | 🔴 | **P0** | [почему критично] |

## Рекомендованный порядок работы

1. **[Модуль A]** — [обоснование]
2. **[Модуль B]** — [обоснование]
...

## План на следующий период

- [...]
```

## Цвета

- 🟢 **Solid** — онтология и инструментарий владеются; только обновление под текущую версию
- 🟡 **Shallow** — умею, но онтологически плаваю; приоритет 2
- 🟠 **Gaps** — куски знаю, цельной картины нет; приоритет 1
- 🔴 **Cold** — почти не работал; полное погружение

## Правила работы с картой

- Карта определяет **приоритет** модулей, а не **наличие/отсутствие**. Все заявленные в программе модули остаются, меняется порядок и глубина.
- Тема 🟢 — быстрое обновление, 1-2 сессии максимум. Не полный модуль.
- Тема 🔴 — полный модуль + возможно дополнительный.
- Пересмотр карты — не постоянный, иначе нет стабильности. Только в явные моменты.

# 5. Meta-Session Records

## Назначение

Записи мета-сессий курса (контур 3). Фиксируют моменты рефлексии траектории и правки методологии.

## Когда заводятся

- Раз в 2 недели (обычная мета-сессия)
- В конце модуля (модульная мета-рефлексия)
- Каждые 3-4 месяца (курсовая рефлексия)

## Формат (обычная мета-сессия)

```markdown
# Мета-сессия #[NN] | [YYYY-MM-DD]

## Период
С [дата] по [дата]

## Факт vs план

### Сессии за период
- Всего: N
- СМД-циклы: N
- Свободная практика: N
- Схемостроение: N
- Экзамены: N

### Claims за период
- Закрыто: N
- Опровергнуто: N
- Зависло в in-progress >2 недель: N (ID: ...)

### Сопоставление
[Совпадает ли с планом прошлой мета-сессии? Где отклонение?]

## Онтология по факту

[Какие темы обсуждались реально? Какие позиции реально использовались?
Совпадает ли с декларацией?]

## Избегаемое

[Какие темы обходились? Какие позиции не занимались? 
Что делалось поверхностно? Это — самый диагностический вопрос.]

## Претензии к методологии

[Что не работает в самом процессе?]

## Решения на следующие 2 недели

- Уровень A: [...]
- Уровень B: [...]
- Уровень C: [...]
- Приоритеты работы: [...]

## Неудобный вопрос от [Методолога]

[Обязательный пункт. Claude должен задать хотя бы один. Записать вопрос и ответ.]
```

## Формат (модульная мета-рефлексия)

```markdown
# Модуль [X] — закрытие | [YYYY-MM-DD]

## Стартовая точка
Что я думал об этой области в начале модуля?

## Текущая точка
Что я думаю сейчас?

## Онтологический сдвиг
[Что стало различать по-другому]

## Позиции, оставшиеся недоступными
[Какие позиции в этой области до сих пор не даются]

## Влияние на следующий модуль
[Что из освоенного меняет подход к следующей теме]

## Оценка проектирования модуля
[Был ли модуль правильно спроектирован? Что переделать?]
```

## Правила

- Записи хранятся в папке `meta-sessions/` внутри курса
- Именование: `ms-NN-YYYY-MM-DD.md` для обычных, `module-X-closeout.md` для модульных
- Каждая запись — отдельный коммит с осмысленным message

# 6. Methodology Journal

## Назначение

**Не в курсе, а в корне методологического корпуса.** Это журнал эволюции самого Method. Отличается от meta-sessions курса — те про конкретный курс, а journal — про методологию как таковую.

Хранится в `smd-learning/methodology-journal.md` (а не в `courses/*/`).

## Когда обновляется

После каждого методологического диалога (в отдельном Project "Methodology Development"), в котором принято решение уровня **D** (фундаментальные принципы). Правки уровней A/B/C фиксируются в соответствующих документах курса через meta-sessions, не в journal.

## Формат записи

```markdown
## [YYYY-MM-DD] — [краткая тема]

### Контекст
Почему этот диалог возник. Какой триггер.

### Ход обсуждения
Ключевые повороты, основные позиции, проблематизации.

### Принятые решения (только уровень D)
- Принцип: [...]
- Обоснование: [...]
- Эмпирическая основа: [какие данные привели — обязательно]
- Commit SHA: [ссылка]

### Отклонённые варианты
Что рассматривали, но решили не делать, и почему.

### Открытые вопросы
Что осталось нерешённым для будущих диалогов.

### Применённые патчи
- `core/methodology-method.md` [коммит]: [что изменено]
- `core/session-patterns.md` [коммит]: [что изменено]
```

## Принципиальное правило

Запись в journal **требует эмпирической основы**. Если правка предлагается "а давай сделаем так, потому что звучит лучше" — запись не делается, решение не фиксируется. Это защита от спекулятивной эволюции.

# 7. Onto-Map (опционально, начиная со второго модуля)

## Назначение

Живая онтологическая карта предметной области — что существует в мире предмета, как связано, какими средствами мыслится.

## Формат

Гибридный — основная часть в markdown (структурированное описание), при необходимости — вложенные mermaid или ссылки на SVG в `schemes/`.

```markdown
# Onto-Map: [предмет]

## Уровни существования

### Уровень 1: [название, например, "IoC-контейнер как единица существования"]
[Описание: что это, как существует, ключевые различения]

### Уровень 2: [...]
...

## Ключевые различения

- [X] vs [Y]: [как отличить, типичные ловушки]
- [A] vs [B]: [...]

## Релевантные позиции

Каждая позиция (из [Оппонент]-воплощений) видит эту онтологию по-своему:
- [Позиция 1]: [что выделяет, что игнорирует]
- [Позиция 2]: [...]

## Эволюция понимания

- v1 (YYYY-MM-DD): [исходная гипотеза]
- v2 (YYYY-MM-DD): [что изменилось после Модуля X]
- v3 (YYYY-MM-DD): [...]
```

## Правила

- Обновляется не на каждой сессии, а в моменты сдвигов (обычно после модульной мета-рефлексии)
- Версии не переписывают друг друга — накапливаются (можно видеть эволюцию)
- Версии живут в git-истории, не в самом файле как список (история — в git log)

# 8. Schemes

## Назначение

Графические схемы, построенные в процессе обучения. Не иллюстрации, а **инструменты мышления**.

## Формат

- Предпочтительно mermaid (можно версионировать в git, читается в preview)
- При необходимости — SVG (сгенерированный в чате и сохранённый)
- Никогда не PNG/JPG без сопровождающего "исходника" (git не видит в них изменений)

## Именование

`schemes/[тема]-v[N].mmd` или `.svg`

Примеры:
- `schemes/ioc-lifecycle-v2.mmd`
- `schemes/http-request-flow-v1.svg`

## Правила

- Старые версии не удаляются. Схема **эволюционирует** через новые версии.
- Каждая версия — отдельный коммит с message вида `scheme: [тема] v[N] — [что изменилось]`
- В начале файла схемы — комментарий с датой и кратким описанием изменений от предыдущей версии

# 9. Active Module

## Назначение

**Фиксированный путь**, по которому Claude Project находит содержимое активного модуля. При переходе между модулями содержимое заменяется на следующий модуль.

## Расположение

`courses/[курс]/active-module.md`

## Правила

- Файл существует всегда, начиная с Модуля 0
- Содержимое — полная копия файла из `modules/module-X-[name].md`
- Замена — Speech Act закрытия одного модуля и открытия следующего
- Замена коммитится с message: `switch active module: M[X] -> M[Y]`

## Почему не символическая ссылка

Символические ссылки плохо работают через git и через Claude GitHub connector. Физический файл проще и надёжнее.

# 10. Согласованность артефактов

Артефакты ссылаются друг на друга. Эти ссылки — часть методологии:

- reflection-log → evidence-ledger (Evidence-претензия в логе даёт ID в ledger)
- evidence-ledger → reflection-log + коммиты (Evidence-ссылка указывает на сессию и код)
- meta-sessions → reflection-log + evidence-ledger (сводка за период)
- meta-sessions → правки в course-mechanics / systemprompt (через коммиты)
- heat-map → программа курса (порядок модулей)
- onto-map → schemes (ссылки на графические схемы)

Разрыв согласованности — признак дрейфа. Можно проверять периодически: например, в мета-сессии курса — "есть ли Evidence-претензии из reflection-log без записи в ledger?".
