# Механика учебного процесса
## Как я живу в методологии "Java + Spring Boot 3 как способ мышления"

---

## 1. Онто-карта системы обучения

Обучение — это живая четырёхсоставная экосистема:

**Я (студент-методолог)** — центральная позиция. Действую, рефлексирую, принимаю решения о курсе. Я — со-конструктор методологии, не её потребитель.

**Claude Project** (веб-чат или Desktop app) — пространство мысли-коммуникации и чистого мышления. Здесь живут позиционные диалоги, проблематизации, построение онто-карты, рефлексия сессий, мета-сессии. Claude играет 5 позиций.

**Claude Code** (terminal или JetBrains/VS Code plugin) — пространство мыследействия с реальным кодом. Работа внутри репозиториев.

**GitHub / локальная FS** — материальное тело курса. Два репозитория, в них живёт всё: код, методология, артефакты обучения.

**Ключевой принцип:** мышление и действие разделены по инструментам, но соединены через GitHub. Claude Project читает репозитории через GitHub connector (по запросу). Claude Code пишет в локальную FS, пуш в GitHub делает синхронизацию.

## 2. Двухрепозиторная архитектура

### `smd-learning` — методологический репозиторий

**Что там:** методология курса, ваша история обучения, рефлексии, evidence, схемы. Личный корпус, растущий через годы и разные курсы.

**Статус:** private (это личная интеллектуальная собственность).

**Структура:**
```
smd-learning/
├── README.md                       # Навигация по корпусу
├── CLAUDE.md                       # Контекст для Claude Code в этом репо
└── courses/
    └── java-spring-2026/
        ├── course-mechanics.md     # Этот документ
        ├── project-systemprompt.md # Копия системного промпта
        ├── active-module.md        # СИМВОЛ АКТИВНОГО МОДУЛЯ
        ├── modules/
        │   ├── module-0-audit.md
        │   └── [module-G-generics.md — появится после M0]
        ├── reflection-log.md       # Лог всех сессий
        ├── evidence-ledger.md      # Реестр свидетельств
        ├── heat-map.md             # Результат Модуля 0 (появится)
        ├── onto-map.md             # Живая онтология (растёт позже)
        ├── meta-sessions/          # Записи мета-сессий
        └── schemes/                # Построенные схемы (SVG, mermaid)
```

### `methodolog` — проектный репозиторий

**Что там:** сквозной Spring Boot проект, над которым идёт практика.

**Статус:** сейчас private, позже может стать public.

**Структура:**
```
methodolog/
├── README.md                       # Базовая навигация проекта
├── CLAUDE.md                       # Контекст для Claude Code в проекте
├── pom.xml                         # Parent POM (multi-module)
├── docker-compose.yml              # PostgreSQL + опциональные сервисы
├── methodolog-core/                # Domain + use cases
├── methodolog-persistence/         # JPA + Flyway
├── methodolog-api/                 # REST + Security
├── methodolog-integration/         # XSD, события
└── assessment/                     # Учебный песочник для Модуля 0
    ├── A1/
    ├── A2/
    └── ...
```

### Почему два репо

- **Разные Bounded Context'ы** (FPF A.1.1): методологический корпус — про мышление; проектный — про код.
- **Разная приватность:** рефлексии всегда остаются с вами, проект может однажды стать public portfolio.
- **Масштабирование:** через полгода-год начнёте второй курс — он ляжет в `smd-learning/courses/[новый-курс]/`, не трогая методологический корпус.
- **FPF Single Source of Truth:** каждый репозиторий — один авторитетный источник для своей области.

## 3. Механизм "активного модуля"

В каждый момент времени в `smd-learning/courses/java-spring-2026/active-module.md` лежит содержимое модуля, которым вы сейчас занимаетесь.

- В начале курса `active-module.md` = копия `modules/module-0-audit.md`.
- После закрытия Модуля 0 и получения heat-map вы выбираете следующий модуль, копируете содержимое в `active-module.md` (старое заменяется).
- Архивные модули остаются в `modules/` для истории.

**Зачем:** Claude Project всегда читает по одному и тому же пути. Ему не нужно переключаться между "какой сейчас модуль загружен в Knowledge". Переключение — это явный Speech Act, который делаете вы через Claude Code.

Команды для Claude Code:
```
"Закрываю Модуль 0, начинаю Модуль G. 
Скопируй содержимое modules/module-G-generics.md в active-module.md. 
Коммит: 'switch active module: M0 -> MG'."
```

## 4. Роли в экосистеме

**Я:** единственный, кто принимает решения. Claude не решает "прошёл тему". Claude не решает "пора переходить". Claude не решает "достаточно понимаю".

**Claude Project:** держатель методологии и симулятор позиций. Не даёт готовых ответов. Маркирует позицию. Проблематизирует. Организует встречу с предметом через задачи. Ценность — в качестве вопросов.

**Claude Code:** исполнитель мыследействия. Не объясняет онтологию (для этого есть Project). Запускает команды, анализирует вывод, делает точечный code review с явной позицией, проводит bytecode-инспекцию, **записывает в артефакты**.

**GitHub:** память системы. Если не зафиксировано в репозитории — не случилось.

## 5. Типовая сессия (СМД-цикл, 90 мин)

**Этап 1. Открытие (5 мин).** 
Открываю Claude Project: "Начинаем сессию [N] по Модулю [X], тема [...]. На прошлой сессии зафиксировали [...]. Цель сегодня: [...]". Claude в [Наставник] читает `active-module.md`, подтверждает контекст, формулирует стартовую задачу.

**Этап 2. Вход через действие (20-25 мин).** 
Задача поставлена. Переключаюсь в Claude Code в репозитории `methodolog` (или `assessment/` в нём). Работаю в коде. Без подсказок. Claude Project молчит. Иду до первого камня или до работающего варианта.

**Этап 3. Остановка и проблематизация (15-20 мин).** 
Возвращаюсь в Claude Project, копирую код или ссылаюсь на коммит. Claude переключается в [Критик]. Вскрывает онтологию решения. Не соглашается быстро.

**Этап 4. Чистое мышление (20-25 мин).** 
Работа со средствами. [Наставник] через вопросы. Часто — построение или доработка схемы (артефакт-визуализация в чате, потом экспорт в `schemes/`).

**Этап 5. Возврат к действию (10-15 мин).** 
Та же задача, новое видение. Claude Code помогает проверить (запустить тесты).

**Этап 6. Двойная рефлексия (5-10 мин).** 
Claude в [Методолог]. Три вопроса. Я отвечаю. Claude формулирует запись для reflection-log. **Переключаюсь в Claude Code, прошу применить запись.** Коммит.

### Продукты сессии

- Код/схема (в репозитории, коммит).
- Уточнённая онто-карта (когда актуально).
- Запись в `reflection-log.md` (через Claude Code).

### Анти-паттерны сессии

- Сессия без мыследействия (чистый чат) — допустима не чаще раза в 2 недели.
- Сессия без проблематизации — красный флаг, на следующей Критик должен быть жёстче.
- Сессия без рефлексии — "сессия не случилась как учебная". Лучше 3 строки, чем ничего.

## 6. Мета-сессия (раз в 2 недели)

**Объект обсуждения — сам курс, не Java.** Пропускать нельзя.

**Формат:** Claude Project, триггер "Мета-сессия". Claude только в [Методолог]. Шесть пунктов:

1. **Что реально происходило 2 недели.** Сколько сессий, сколько closed claims (по ledger). План vs факт.
2. **Онтология курса по факту.** Совпадает ли с декларированной?
3. **Белые пятна и тени.** Каких тем/позиций избегаю?
4. **Претензии к методологии.** Что не работает? Какие артефакты избыточны/недостаточны?
5. **Следующие 2 недели.** Приоритизация на основе heat-map и ledger.
6. **Фиксация.** Запись в `meta-sessions/ms-[NN]-[дата].md` через Claude Code.

Claude **обязан** задать минимум один неудобный вопрос — "а действительно ли ты X?". Защита от комфортного самообмана.

## 7. Закрытие модуля (экзамен) — защита перед комиссией

Модуль не закрыт, пока не сдан экзамен. Жёсткая дисциплина Evidence.

**Формат:** 90-минутная сессия. Задача от [Симулянт-эксперт] (senior tech lead) из индустриальной практики, **не похожая** на упражнения модуля. 60 минут на решение в Claude Code в `methodolog`. Затем — последовательная защита перед Советом мудрецов.

**Совет мудрецов** — четыре эксперта по сквозным темам курса, каждый задаёт 1–2 вопроса из своей линзы:

1. **Страж AOP/Proxy** — proxy-механика, self-invocation, runtime-поведение аспектов
2. **Страж Testing** — тестовая стратегия, уровни тестов, хрупкость
3. **Страж SB3** — идиоматичность для Spring Boot 3, использование новых возможностей
4. **Страж Clean Architecture** — слои, зависимости, DIP, Clean Code принципы

Защита последовательная: эксперт атакует → учащийся защищается → следующий эксперт. Формат защиты курсовой в МФТИ.

**Три исхода:**

- **Passed at Constructive**: модуль закрыт, ledger updated. Комиссия единогласна.
- **Passed at Transfer**: закрыт *условно*. Один или два эксперта поставили concern. В ledger — "закрытие с долгом", вернуться к слабым местам за 2 недели.
- **Not passed**: модуль не закрыт. Два и более concern с серьёзными пробелами. Анализ причин, возврат к конкретным темам.

**Нельзя:** "устал, давай дальше". Лучше задержаться, чем накапливать долг.

## 8. Формат артефактов

### `reflection-log.md` — лог сессий

Записи от новых к старым. Формат строгий (Claude Project опирается на него):

```markdown
## Сессия #23 | 2026-04-20 | Модуль 5 | Тема: N+1 и EntityGraph

### Контекст
[1-2 предложения: откуда пришли]

### Позиции в сессии
- Вход: Наставник → Симулянт (tech lead)
- Основная работа: Критик (~25 мин), Наставник (~15 мин)
- Закрытие: Методолог

### Сдвиг содержания
[Что новое теперь понимаю — 2-4 предложения, формулировка сдвига, НЕ список тем]

### Сдвиг процесса
[Что заметил о собственном обучении — 1-2 предложения]

### Открытые вопросы
- [ ] Вопрос 1
- [ ] Вопрос 2

### Evidence-претензия
[Коммит, тест, файл схемы или "не готов предъявить"]

### Артефакты
- code: methodolog@a3f7c2b
- scheme: schemes/bean-lifecycle-v2.svg (если построена)
```

### `evidence-ledger.md` — реестр свидетельств

```markdown
| ID | Модуль | Claim | Level | Evidence | Дата | Статус |
|----|--------|-------|-------|----------|------|--------|
| 042 | M5 | Понимаю LAZY/EAGER и runtime-последствия | T | methodolog@a3f7c2b | 2026-04-20 | ✅ closed |
| 043 | M5 | Могу диагностировать N+1 и применить @EntityGraph | C | exam M5 | TBD | 🟡 in-progress |
```

- **Level:** R (Reproductive), T (Transfer), C (Constructive).
- **Статусы:** ⬜ planned, 🟡 in-progress, ✅ closed, ❌ falsified (claim был, Evidence не подтвердилось — важнейшая категория честности).

### `heat-map.md` (после Модуля 0)

Каждая тема курса получает цвет:
- 🟢 **Solid**: онтология и инструментарий. Обновление под SB3/J21.
- 🟡 **Shallow**: умею, но плаваю онтологически. Приоритет 2.
- 🟠 **Gaps**: куски знаю, цельной картины нет. Приоритет 1.
- 🔴 **Cold**: почти не работал. Полное погружение.

Определяет порядок модулей индивидуально, а не канонически.

## 9. Сигналы и красные флаги

**Флаг 1: "Сессии слишком гладкие"** — 3 сессии без "стоп, не понимаю". Проблематизация сломалась. Действие: попросить Критика жёстче.

**Флаг 2: "Reflection-log копипаста"** — записи похожи, сдвиг стал штампом. Ставить как главный вопрос ближайшей мета-сессии.

**Флаг 3: "Claims не закрываются"** — много 🟡, мало ✅. Либо слишком амбициозные, либо избегание. Ревизия claims.

**Флаг 4: "Код работает, а понимания нет"** — прошёл экзамен, но обоснование плывёт. Evidence понижается с C до R. Проблематизация решения через неделю.

**Флаг 5: "Пропустил мета-сессию"** — 3+ недели без неё. Сопротивление мета-работе. Провести немедленно с вопросом "почему избегал?".
