### Позиции в сессии
- Вход: Методолог (калибровка, протокол)
- Основная работа: Симулянт-эксперт (выдача задачи) → Критик (вскрытие онтологии) → Наставник (compact constructor) → Критик (второй submission)
- Закрытие: Методолог (финальная рефлексия)

### Сдвиг содержания
Раньше я склеивал в одно понятие "тип значения" несколько разных: DDD Value Object, C# struct, JVM value types (Valhalla). Теперь различаю: моя интуиция "объект без независимой identity, различимый только по содержанию" — это value-based class / nominal tuple. Record — его реализация. Термин "value type" в JVM-контексте про другое. Отдельно: понял, что контракт equals/hashCode опирается ровно на "только поля самого класса" — это источник семантики равенства, не техническая деталь.

### Сдвиг процесса
Назвал два конкретных отсутствующих рефлекса (заменили размытое "невнимательность"):
1. REPL-стиль работы — писать → запустить → проверить, а не "написал и сдал".
2. Fix one → check all — после нахождения ошибки в одном месте проверять распространение в коде.
Плюс: различил "активный словарь" (которым пишу) и "знаемый словарь" (которым спорю) — первый обновляется только через письменное воспроизведение, не через чат.

### Открытые вопросы
- [ ] Проверить в боевом коде, сколько мест со сломанным equals/hashCode из-за `super.xxx` от Object (оценка: высокая вероятность).
- [ ] Когда появляются реальные вопросы про records — через день, через неделю? Если нет — сигнал, что задача не "осела".
- [ ] Как организовать REPL-стиль в Java (jshell? IDE-scratch files?) в привычку.
