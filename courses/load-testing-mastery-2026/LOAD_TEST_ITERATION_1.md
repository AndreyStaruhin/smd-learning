## 1. Цель итерации

Найти **стационарный предел пропускной способности нижней части системы digitalprofile.individuals** при двух конфигурациях воркеров `queue-person-executor`: **N=5** и **N=10**. Результат — материал для разговора с командой о выборе финальной конфигурации воркеров к ближайшему релизу.

Тип итерации — **рабочий**, не разведочный. Результат должен быть в форме, переносимой в команду (см. §9).

## 2. Объект измерения и точка инжекции

### 2.1 Объект измерения

Измеряются как одна подсистема:

- HTTP-обработчик входящих сообщений реверс-СМЭВ-API;
- создание задачи в `tasks` через `StartScenarioEvent` → `TaskReadyToCreateEvent`;
- воркеры (`PersonQueueTaskHandler`, `queue.executor.threadPoolSize=N`);
- PostgreSQL (выборка через `SELECT FOR UPDATE SKIP LOCKED`, INSERT в `tasks`/`history`).

### 2.2 Точка инжекции — реверс-СМЭВ-эндпоинт

«Реверс-СМЭВ» — историческое название направления **«гейт → наш сервис»** через TLS HTTP. Появилось, когда направление обращения «развернулось»: раньше наш сервис сам отправлял запросы в гейт, теперь гейт отправляет нам. Технически это входящий HTTP-API сервиса под `/smev/api/v1/*`.

**Кандидат-эндпоинт по умолчанию:** `POST /smev/api/v1/PersonalData` (см. также открытый вопрос §13.10 про альтернативу `/CustomerPermissionList`).

| Поле | Значение из кода |
|---|---|
| URL | `/smev/api/v1/PersonalData` |
| Класс/метод обработчика | `PersonsSmevController.receivingPersonalDataDocuments()` (`PersonsSmevController.java:56-87`) |
| Формат тела | **JSON** (контроллер настроен с `consumes = APPLICATION_JSON_VALUE`) |
| DTO тела | `ReceivingPersonalDataDocWithScopeSmev3Request` (с `@Valid`) |
| Обязательные заголовки | `X-Bank-Id` (UUID), `X-Request-Id` (String) |
| Опциональные заголовки | `X-Repository-Id` (String), `X-Protocol-Type` (enum `ProtocolType`) |
| Тип задачи в очереди | `taskType=person` (через `DataType.RECEIVING_PERSONAL_DATA_DOC_REQUEST` → `QueueTaskTypeMapper.map(...)`) |

### 2.3 Что обработчик делает синхронно ДО возврата 200 OK

Из кода `SmevService.handle()` (вызывается из `PersonsSmevController:81`):

1. `sendAdapterRequest(...)` — **исходящий SOAP-вызов в СМЭВ-адаптер** через `adapterRequestExecutor.sendRequest(...)` (на стенде — WireMock-стаб, задержка ~150 мс).
2. `prepareScenarioData(...)` — маппинг в `GetResponseStageData`.
3. `createSmevScenario(...)` → `publisher.publishEvent(StartScenarioEvent)` → синхронный `@EventListener` `ScenarioEventService.handleStartScenarioEvent` → `publisher.publishEvent(TaskReadyToCreateEvent)` → запись задачи в `tasks` (через handler-listener).
4. **Только после этого** контроллер возвращает `200 OK`.

> **Наблюдение Claude Code (фактчекинг):** на каждый POST реверс-СМЭВ-эндпоинт сервис **синхронно** делает один SOAP-вызов в СМЭВ-адаптер (на стенде — WireMock ~150 мс). Затем создаётся задача со стадией `GetResponse`, которую воркер обработает асинхронно — и в стадии `GetResponse` будет ещё один SOAP-вызов в адаптер. То есть один POST → **два WireMock-вызова**: первый синхронно во входе, второй в воркере (на разных шагах жизненного цикла). Этот факт является **открытым вопросом для интерпретации** результатов (см. §13.6).

### 2.4 Что нужно мокировать

Помимо уже принятых WireMock-стабов из `LOAD_TEST_PLAN.md` §3 (СМЭВ-адаптер на `SmevOptions.AdapterUri`, Gate-out на `Gates.AbsGateUri`) — **дополнительных стабов не нужно**. Внешний ТР ЕСИА в этом сценарии не задействован: цепочка `/cb` не используется, аутентификация пользователя не происходит.

### 2.5 Минимально необходимые поля тела

Тело должно проходить `@Valid`-проверки `ReceivingPersonalDataDocWithScopeSmev3Request`. Конкретный набор обязательных полей зафиксирован Bean Validation-аннотациями этого DTO. Шаблон фиксированный — мы не тестируем валидацию.

**Открытый вопрос (см. §13.7):** какой минимально-валидный JSON, проходящий все `@Valid`-проверки? Это вопрос **составителя Gatling-сценария на следующем шаге**, не часть этого плана.

## 3. Claim Scope результата

Результат относится к **нижней части системы**. Узкое место в верхней части (`/cb`-цепочка, реальная аутентификация ТР ЕСИА, валидация `customerUuid` в `CbEsiaService.validateCustomerUuid`) этим тестом **не находится**.

**Не входит в критерий приёмки этой итерации:**

- **Латентность отдельной задачи в очереди** (`task_latency_in_queue`). Если бизнесу окажется важна — отдельная итерация.
- **Burst-сценарии** (способность переваривать пики). Меряем только **стационарную пропускную способность**.
- **Сравнение с REST-сценарием A или комбинированной нагрузкой C** (см. base plan §5, §6.1).
- **USE-метрики ресурсов** (CPU, GC, Hikari) — не входят в первичное решение по итерации (см. §7.1, §13.4).

## 4. Профиль инжекции

Ступенчатый ramp-up. Шаг ступени привязан к теоретическому потолку throughput.

**Расчёт теоретического потолка:**

- WireMock-задержка на внешний вызов = 150 мс.
- Оценочное время обработки одной задачи воркером ≈ 200 мс _(подлежит проверке — см. §4.1)_.
- Теоретический потолок при 5 воркерах ≈ **25 RPS**.
- Теоретический потолок при 10 воркерах ≈ **50 RPS**.

| Конфигурация | Шаг ступени | Ступени RPS | Длительность плато |
|---|---:|---|---:|
| **N=5** воркеров | 5 RPS | 5 → 10 → 15 → 20 → 25 → 30 | 2 мин (ориентир) |
| **N=10** воркеров | 10 RPS | 10 → 20 → 30 → 40 → 50 → 60 | 2 мин (ориентир) |

**Переходы между ступенями** — несколько секунд (не мгновенно), чтобы избежать сетевых артефактов на API.

**Длительность плато 2 минуты — первая попытка**, корректируется на первом прогоне по форме реальных графиков.

### 4.1 Открытый вопрос: точное `T_task`

Допущение `T_task ≈ 200 мс` требует сверки по коду.

**Из кода видно**, что обработка одной задачи стадии `GetResponse` воркером включает:

- `taskRepository.getNextPerson(...)` — SQL `SELECT FOR UPDATE SKIP LOCKED` + перевод задачи в `running=true` (`PersonQueueTaskHandler:51`).
- Десериализация `data.jsonData()` в `GetResponseStageData` (Jackson, `ScenarioEventService.handleTaskReceivedEvent`).
- `scenario.executeStage(...)` для стадии `GetResponse` — внутри **исходящий SOAP-вызов в адаптер** (WireMock ~150 мс), плюс анализ ответа.
- Публикация результата (`TaskCompletedEvent` / `TaskExecutedWithTransientErrorEvent` / ...).
- SQL смены статуса в `tasks` + INSERT в `history`.

**Дополнительно** см. §2.3: сам POST на реверс-СМЭВ-эндпоинт уже сделал один WireMock-вызов синхронно во входе **до** создания задачи. То есть лимитирующая операция «WireMock 150 мс» встречается дважды на «POST → задача обработана», но в **разных** местах жизненного цикла (HTTP-вход и воркер).

Является ли `T_task ≈ 200 мс` корректной оценкой для нашего реального жизненного цикла или фактическое значение существенно другое — **открытый вопрос (см. §13.1)**, проверяется на warm-up первого прогона.

## 5. Первичный маркер останова ступени

**Метрика:** `dp_queue_tasks_backlog_size{taskType="person"}` (Gauge — спецификация в `LOAD_TEST_PLAN.md` §7).

### 5.1 Критерий стационара (визуальный, первая попытка вербализации)

На каждой ступени оцениваются **две фазы**:

1. **Переходная** — после повышения темпа RPS, backlog растёт (или резко скачет).
2. **Стационарная** — после выхода на новый уровень.

**Стационар признаётся:** на стационарной фазе средняя линия backlog **не возрастает** относительно первой минуты этой фазы (визуально, без формального порога).

**Стационар не признаётся:** на стационарной фазе средняя линия продолжает расти, новое плато не достигается до конца ступени.

### 5.2 Вердикт по конфигурации

> Предельный RPS = последняя ступень, на которой стационар признан.

**Точность результата** = шаг ступени (5 RPS для N=5, 10 RPS для N=10).

### 5.3 Открытый вопрос: формальный критерий стационара

Окно наблюдения и порог скорости роста для формального критерия **не сформулированы**. Разрабатываются на основе наблюдений первого прогона и **переходят в следующую итерацию**.

В рамках этой итерации критерий — **визуальный**, его выносит исполнитель прогона на дашборд Grafana.

## 6. Вторичные маркеры

Используются для **интерпретации результата**, не для принятия решения «прошла ли ступень».

### 6.1 Ошибки на входе HTTP реверс-СМЭВ

```promql
http_server_requests_seconds_count{uri=~"/smev/api/v1/.*",status=~"4..|5.."}
```

с группировкой/фильтром по `status` и `uri`. Это стандартная Spring Actuator метрика — отдельных кастомных метрик не требует.

### 6.2 Ошибки в обработке (внутри воркеров)

```promql
dp_queue_tasks_processed_total{result="transient_error",taskType="person"}
dp_queue_tasks_processed_total{result="fail",taskType="person"}
```

(спецификация метрики в `LOAD_TEST_PLAN.md` §7).

### 6.3 Правила интерпретации

Графики выводятся **раздельно**. **Не суммируются** — это разные физические механизмы (насыщение верхней vs нижней части трубы).

### 6.4 Открытый вопрос: пороги допустимой ошибки

Пороги «допустимой» ошибки на каждом графике в первой итерации **не выбраны**. Записывается **фактический** error rate, без приёмочных порогов.

## 7. Метрики дашборда первой итерации

Минимально достаточный набор, отвечающий на вопрос «стационарен ли backlog на ступени и не появились ли ошибки».

| Назначение | Метрика / выражение | Тип | Источник |
|---|---|---|---|
| Первичный график | `dp_queue_tasks_backlog_size{taskType="person"}` | Gauge | base plan §7 (кастомная) |
| Вторичный 1 (ошибки на входе) | `rate(http_server_requests_seconds_count{uri=~"/smev/api/v1/.*"}[1m])` by `status` | стандартная Spring | Spring Actuator |

### 7.2 Метрики из base plan §7, которые НЕ входят в дашборд первой итерации

Эти метрики **существуют** в спецификации base plan §7, но в первой итерации **не входят в дашборд решения**:

- `dp_queue_tasks_incoming_total` — потенциально полезен для расчёта `backlog_growth = incoming − processed`, но первичный маркер этой итерации — **абсолютный backlog**, не разница потоков. Использование — открытый вопрос для следующих итераций.

## 8. Tear-down между прогонами (открытый вопрос)

В рамках этой итерации проводится **минимум два прогона** — N=5 и N=10. Состояние между ними нужно приводить к одной точке. Решение **не принято в M1.4**, ниже — **развёрнутый список альтернатив без выбора**:

| Вариант | Что делает | За | Против |
|---|---|---|---|
| **A.** TRUNCATE `tasks`, `history`, `esia_operations` | Очистка таблиц очереди и операций | Чистая БД, повторяемость | Не убирает heap, JIT-кеши; `pg_stat_statements` уже «прогрет» |
| **B.** Полный рестарт сервиса | Сброс JVM | Чистый JIT, heap, метрики Micrometer; **необходим в любом случае** для смены `queue.executor.threadPoolSize` (читается на старте) | Требует прогрева перед прогоном |
| **C.** Прогрев перед прогоном | Sanity-warm-up на низкой нагрузке | Стабильные первые ступени | Удлиняет цикл; форма прогрева ещё не выбрана |
| **D.** Чистка WireMock state | Сброс `request journal` | Без ложных «совпадений» предыдущего прогона | Зависит от конфигурации стабов; может не понадобиться |
| **E.** Никакого tear-down | Прогоны идут подряд | Быстро | Эффект «прогретой БД» / накопления данных может отличать N=5 от N=10 не из-за воркеров |

**Решение принимается на встрече с командой.** В отчёте первой итерации фиксируется фактически использованный вариант — для воспроизводимости.

## 9. Формат отчёта первой итерации

Per-конфигурация (отдельно для N=5 и для N=10):

- **Предельный RPS = X** (последняя пройденная ступень).
- **Что произошло на следующей ступени** — рост backlog без стабилизации, появление ошибок какого типа, на каких метриках это видно.
- **График backlog** за весь прогон, с разметкой ступеней.
- **Графики двух error-метрик** за весь прогон (раздельно, см. §6).

В Claim Scope результата явно записано:

- что измеряли (нижняя часть системы);
- что **не** измеряли (`/cb`-цепочка, ТР ЕСИА, латентность задачи в очереди, burst);
- точность ответа = шаг ступени.

## 10. Расхождения с LOAD_TEST_PLAN.md

Перечень ниже — для **отдельной работы автора плана с командой**, не разрешается в этой итерации. Этот документ — план одной итерации, не пересмотр общего плана.

### 10.1 `queue.executor.threadPoolSize` — фактическое значение в master

`LOAD_TEST_PLAN.md` §2 указывает «`queue.executor.threadPoolSize=10`». Фактическое значение в `core/src/main/resources/application.yaml` master-ветки на момент составления этого плана — **`10`** (строка 51).

В исходной формулировке решений (M1.4) автор плана упоминал «текущее прод-значение, по словам автора, 5 (с возможной корректировкой)». Возможные расхождения:

- master vs прод-окружение могут отличаться;
- обсуждаемое значение «5» — возможно, целевая конфигурация, к которой команда планирует прийти, а не текущая прод-настройка.

**Открытый вопрос (см. §13.8):** какое значение `threadPoolSize` стоит в проде сейчас, и какая из двух конфигураций (N=5 / N=10) ближе к текущему прод-состоянию? Это нужно для интерпретации «что значит наш результат относительно прода» в финальном отчёте.

### 10.2 Реверс-СМЭВ как точка инжекции — пересмотр scope

В более ранних редакциях `LOAD_TEST_PLAN.md` «реверс-СМЭВ» исключался из scope. После уточнения терминологии в base plan §1 (реверс-СМЭВ = входящий HTTP-API «гейт→нас», не пуллер нотификаций) реверс-СМЭВ-API **в scope** общего плана и используется в сценариях B-HTTP и C. Эта итерация делает реверс-СМЭВ-API **точкой инжекции**.

**Причина первоначального исключения** реверс-СМЭВ из scope в более ранних редакциях не зафиксирована. **Открытый вопрос (см. §13.9):** автору base plan восстановить причину для архивной справки — была ли это терминологическая ошибка, или существовало содержательное возражение, которое мы упустили.
