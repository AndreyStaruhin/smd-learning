# 1. Профиль учащегося (финализированный)

- **Роль относительно продукта `digitalprofile.individuals`:** разработчик проекта. Имеет доступ к кодовой базе, может вносить PR (в частности, добавление `micrometer-registry-prometheus` в `core/pom.xml`, настройку actuator endpoints в `application.yaml`).
- **Java-опыт:** около 1 года, самооценка — middle уровень.
- **Предыдущий стек:** .NET Core 3.1 (разработка). Это значит: базовая онтология HTTP-сервисов, пулов, ORM, GC — знакома по .NET, требует адаптации терминов и специфики JVM.
- **Observability-опыт:** Prometheus/Grafana — «раз в пару лет, гуглю и настраиваю под задачу». Micrometer — не трогал. Уровень онтологии observability как системы — низкий.
- **Релизный цикл продукта:** **ежемесячный**. Это кардинально меняет форму курса по сравнению с шаблонным on-ramp (см. §2).
- **Параллельные курсы:** идёт курс «Java/Spring-мышление» по той же методологии, в отдельном Claude Project. Начал, активно движется. На пересечения отвечает сам учащийся (см. §7).

# 2. Отклонения от шаблона `course-template.md`

Эти правки уровня C (программа курса) приняты на первой аудитной сессии. Все обоснованы и зафиксированы.

## 2.1. Сокращённый Модуль 0

**Стандарт:** 5 сессий ассессмента с практическими задачами на R/T/C + финальная сборка heat-map.

**У нас:** одна аудитная сессия (уже проведена) + первичная heat-map на основе самооценки учащегося. Без практического ассессмента.

**Обоснование:** учащийся сам оценил свой уровень как near-zero в load testing. Детальный ассессмент через практические задачи показал бы то же самое, что показала беседа, но съел бы 2 недели из 4 недель первого релизного цикла. Heat-map **будет перепроверяться** через реальную работу в Модуле 1 — если что-то выстрелит не так, как ожидалось, правим.

**Риск этого решения:** возможна недооценка или переоценка по темам, которые сейчас стоят 🟠 Gaps. Компенсация: первые 2 сессии Модуля 1 идут с повышенным вниманием к heat-map — если реальность расходится с самооценкой, фиксируем правку немедленно, не ждём мета-сессии.

## 2.2. Сжатый on-ramp

**Стандарт:** `course-template.md §4` — первые 2 недели на установку учебной дисциплины, запрет режима цейтнота.

**У нас:** учащийся уже в методологии (параллельный курс Java/Spring по той же методологии). Дисциплину СМД-цикла, двойной рефлексии, Evidence-ledger устанавливать с нуля не нужно.

**Правка:** запрет режима цейтнота в первые 2 недели **сохраняется** (это защита от съезжания в тактику с первого дня), но сама дисциплина считается установленной. Это значит: первая учебная сессия (Модуль 1, Сессия 1) может быть полноценным СМД-циклом без адаптационных послаблений.

## 2.3. MVP первого подхода

**Стандарт:** `course-template.md` не регламентирует «MVP подхода» — там предполагается свободное планирование модулей по heat-map.

**У нас:** ежемесячный релизный цикл требует явной формулировки, что считается «значимым результатом» на каждом подходе. Без этого первый же месяц расплывётся.

**MVP первого подхода (M1):**

1. Docker-compose стенд на локальной VM: сервис `digitalprofile.individuals` + PostgreSQL + WireMock + Prometheus + Grafana. Воспроизводимость — одной командой.
2. PR в продукт: `micrometer-registry-prometheus` в `core/pom.xml`, настройка actuator endpoints (только на стенде, не на master).
3. Минимальный Gatling-сценарий A (Java DSL) — ОДИН прогон, baseline. Не ramp-up, не поиск предела. Просто: фиксированный профиль N виртуальных пользователей, стабильное 5-10 минут, получили цифры.
4. Один читаемый отчёт: p50/p95/p99 на GET `/cb`, error rate, и три-четыре ключевых метрики из Grafana (JVM heap, GC pauses, Tomcat threads, Hikari active). С текстовым описанием: что видно, что непонятно.

**Явно НЕ входит в MVP первого подхода:**

- Сценарий B (нагрузка очереди) — переносится на M2 или M3.
- Поиск предела через ramp-up — переносится.
- Полный набор панелей Grafana — переносится.
- CI-интеграция — переносится.
- Работа с `SKIP LOCKED` — переносится (связано со сценарием B).

**Принцип:** «что-то работающее и читаемое» важнее «всеобъемлющее и сырое». Проверяется в мета-сессии после релиза — не деградировал ли принцип в «что угодно, лишь бы быстрее».

# 3. Стек и инструменты (финализировано)

- **Gatling DSL:** Java. Обоснование — основной язык учащегося, команда на Java, избегание четвёртого нового инструмента (Gatling + Prometheus + Micrometer + Scala/Kotlin — перегруз). Gatling 3.7+.
- **Build system:** Maven (совпадает со стеком продукта — `core/pom.xml`).
- **Grafana dashboards:** на старте ручные, потом — as-code (JSON в git). Решение о as-code принимается после первого прогона.
- **Prometheus:** scrape-интервал 15 сек (стандартно для initial setup; пересматривается, если потеряем сигнал).
- **WireMock:** standalone-сервис в docker-compose; stubs — в проектном репо `digitalprofile.loadtests` как JSON-файлы.
- **Репо для тестов:** `digitalprofile.loadtests`, отдельное от продукта и от `smd-learning`.

# 4. Инстанциация позиций Claude

## 4.1. Активные воплощения [Оппонент]

На первой сессии из четырёх контр-парадигм системного промпта отобраны **две первичных** — остальные активируются, если возникнут естественно:

1. **«Synthetic load is fiction»** — сторонник production shadow traffic. Релевантен для учащегося, потому что сервис интегрирован с ТР ЕСИА, и реальный паттерн нагрузки (распределение `customerUuid`, длительность сессий, доля fail-пути) на стенде не воспроизводится. Эта позиция будет бить в болевую точку «локальный vs Enterprise-тест».
2. **«Continuous performance»** — SRE, считающий, что ad-hoc перед релизом = антипаттерн. Релевантна, потому что у учащегося именно ежемесячный релизный цикл — это идеальная среда для CI-интеграции, которую мы сознательно отложили на потом. Эта позиция будет проверять решение отложить.

Остальные две (chaos over load, code over DSL) — в резерве, вводятся при необходимости.

## 4.2. Активные воплощения [Симулянт-эксперт]

**Первичные роли для Модуля 1:**

1. **Performance Engineer** — думает в терминах latency budgets, percentiles, Little's Law. Основная роль экзаменатора в конце M1.
2. **SRE** — думает в терминах Golden Signals, observability. Вторая роль для обсуждения дашбордов.
3. **Capacity Planner** — в резерве, ближе к концу M1 или M2.

Queue/Async Systems Engineer и DBA добавятся, когда дойдём до сценария B.

## 4.3. Ключевые опорные точки для [Критик]

Из болевых точек учащегося, названных на аудите:

- **Опора 1:** «сколько запросов в секунду выдерживает». Критик бьёт именно в эту формулировку — всегда спрашивает «что значит *выдерживает*? какой профиль? при каких условиях? какой p99?». Задача — чтобы учащийся **перестал давать прямой ответ** на такие вопросы без встречных уточнений.
- **Опора 2:** «локальный тест vs Enterprise». Критик спрашивает про валидность переноса: «эта цифра, полученная на твоей VM, что говорит про прод? какие допущения в переносе? где допущения ломаются?». Это про Claim Scope каждого результата.

# 5. Специфика длительности сессий и ритма

- **Стандартная СМД-сессия:** 90 минут как в ядре. Сжатие не практикуется.
- **Недельная плотность:** 2–3 СМД-сессии в неделю. **Меньше двух — сигнал**, что курс проваливается; больше четырёх — риск выгорания, учитывая что учащийся параллельно ведёт Java/Spring-курс по той же методологии.
- **Мета-сессия:** одна за подход (за месяц) — обязательна, проводится после релиза, перед началом следующего подхода. Если по итогам первого подхода станет ясно, что нужна чаще — пересматриваем.
- **Модульная мета-рефлексия:** совпадает с мета-сессией подхода (один модуль = один подход, по MVP-принципу).

# 6. Правило Evidence при пересечении с параллельным курсом

Учащийся параллельно идёт по курсу «Java/Spring-мышление». Есть пересечения предметных областей (Hikari, Tomcat threads, JPA/Hibernate, bean lifecycle, JVM/GC).

**Правило:** в ledger нашего курса Evidence закрывается **только по задачам нашего курса**. Знание, принесённое из Java/Spring-курса, не даёт автоматического закрытия claim здесь. Оно **ускоряет** работу, но не заменяет её.

**Механика пометки пересечений:**

- Когда Claude Project ловит пересечение, он явно маркирует реплику тегом `[пересечение с Java/Spring-курсом]`.
- Учащийся отвечает: «знаю оттуда, применяю здесь» или «пока не проработано, берём минимум для load testing».
- В ledger — отдельная колонка `cross-course-notes` для темы, если она пересекается. Формат записи: `java-spring-course: 🟢 M3`, или `java-spring-course: not-yet-covered`.

**Почему это важно:** без этого правила через 3 месяца в нашем ledger-е будут «закрытые» темы, которые учащийся на деле не проверял под нагрузкой. Это размывание Evidence-дисциплины, защищённое ядром методологии.

# 7. Специфика режима цейтнота в этом курсе

Базовые правила — в системном промпте («Режим цейтнота»). Курсовые дополнения:

- **Первые 2 недели — цейтнот запрещён.** Стандартное правило ядра, подтверждается.
- **Во всём первом подходе (первый месяц) — цейтнот только через явный Speech Act учащегося**, даже после окончания первых 2 недель. Это защита: у нас MVP и так минимальный, цейтнот здесь означает «делаем тяп-ляп», а не «экономим время на второстепенном».
- **Делегация инфраструктурных задач (docker-compose, Prometheus scrape-config) допустима в цейтноте начиная с 3-й недели**, при условии мини-рефлексии «что я не понял в том, что Claude сделал за меня».
- **Делегация написания Gatling-сценариев запрещена во всём первом подходе**. Это основной предмет курса, его делегация обессмысливает курс.
