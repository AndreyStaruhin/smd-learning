### Сдвиг содержания
Собран план первой итерации первого подхода. Принципиальные решения:
- **Тип итерации — рабочий** (не разведочный): результат должен быть в форме, переносимой в команду.
- **Объект измерения — нижняя часть системы** (HTTP-вход реверс-СМЭВ + воркеры + БД), без `/cb`-цепочки и реальной аутентификации ТР ЕСИА.
- **Точка инжекции — реверс-СМЭВ-эндпоинт.** Это пересмотр scope относительно `test-plan §2` (где реверс-СМЭВ был out-of-scope). Причина первоначального исключения требует восстановления автором плана.
- **Профиль инжекции — ступенчатый ramp-up.** Шаг ступени привязан к теоретическому потолку throughput из системы (WireMock 150 мс × число воркеров), не к round number.
- **Первичный маркер — backlog_size**, критерий стационара визуальный (первая попытка вербализации). Формальный критерий — открытый вопрос, переходит в следующую итерацию.
- **Вторичные маркеры — два потока ошибок** (HTTP на входе и `transient_error|fail` на обработке), наблюдаются раздельно.
- **Формат результата:** число «предельный RPS = последняя пройденная ступень», точность = шаг ступени, для двух конфигураций воркеров (N=5 и N=10).

Применение M1-003 (различение трёх фреймворков RED/USE/Golden Signals) проявилось в работе по разделению backlog (Saturation очереди) и CPU/RAM (USE по ресурсам) как разных осей наблюдения, не сводимых друг к другу. Не закрытие R, но кандидатный материал.

Главное содержательное различение сессии: **«предельный RPS» расщеплён на два смысла** — темп системы целиком (incoming ≤ processing) и судьба отдельной задачи (latency in queue). Учащийся выбрал первый смысл, второй — явно отложил в Claim Scope как сознательно невходящий в первую итерацию.

### Сдвиг содержания
Различены три фреймворка как **три позиции наблюдения**:
- RED — взгляд снаружи, как видит клиент (Rate / Errors / Duration).
- USE — взгляд изнутри, по ресурсам (Utilization / Saturation / Errors).
- Golden Signals — гибрид: пользовательский взгляд + один индикатор насыщения (Latency / Traffic / Errors / Saturation).

Связь с парадигмами:
- **Парадигма A (REST-RPS)** — RED применим напрямую, потому что есть «один клиент» (Gatling), который видит время отклика, ошибки, темп.
- **Парадигма B (queue-throughput)** — RED **расщепляется**:
  - Duration → три разных времени (latency in queue / processing time / end-to-end).
  - Errors → становятся приватными (происходят внутри воркера, снаружи не видны), требуют **кастомных метрик** на смену состояния задачи.
  - Rate → расщепляется на incoming / processing / backlog (как функция от первых двух).

Из этого — онтологическое решение: для парадигмы B стандартных шести семейств Micrometer недостаточно, нужны кастомные доменные метрики для бизнес-сущностей (задача, статус, переход состояния, обращение к внешней системе).

Различение «нагрузочное тестирование vs дебаггинг под нагрузкой»: htop + логи метода + сопоставление графиков — это диагностика постфакт, метод расследования. Нагрузочное тестирование — прогностический метод под контролируемой нагрузкой. Разные вопросы, разные метрики. Для разработчика, который **и тестирует, и чинит**, это два слоя одного процесса, не конкурирующие методы.

**Тип:** C-вопрос (онтологический) — ставит под сомнение само допущение, что «скорость» и «глубина» — это независимо настраиваемые параметры, а не пара в фундаментальном напряжении.

**Заметка для будущих сессий:** этот вопрос неизбежно вернётся в последнюю неделю первого подхода (за неделю до релиза, когда часть MVP будет не готова). Сейчас зафиксирован как опорная точка на тот момент.
