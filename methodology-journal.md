## 2026-05-04 — Сессия M1.9, гипотеза knowledge-digest, кандидат A-правки инструкций MD (исследовательский синтез)

### Контекст

Ревизия сессии M1.9 курса Load Testing в проекте Methodology Development.
Учащийся принёс четыре замечания по сессии:

2. Обобщение #1 — «нужно лучше осведомляться, где мы сейчас» (на старте,

   делает Claude Code», на фоне зафиксированного расхождения с фактом —

### Ход обсуждения

Замечания обработаны по уровням коррекции с явным разнесением «симптом /
кандидатные диагнозы / решение».

По #1 — установлен сбой инстанциации (нарушение чек-листа «Начало
диалога» в курсовом системном промпте — не открыта heat-map по теме
сессии до постановки задачи), а не сбой ядра. Триггер β подрежима в
`core/session-patterns.md` существует, но реактивен по природе.
Эмпирика для B-правки ядра (введение явного шага Knowledge-Readiness
Check в Этап 1) — недостаточна (один случай).

По #2 — гипотеза о расширении дисциплины [Методолога] (проверка
knowledge-readiness не только в Этапе 1, но и в Этапе 3, и при
активации [Симулянт-эксперт]). Эмпирика — три точки в одной сессии.
Не оформляется ни на каком уровне; копится для мета-сессии после
первого релиза курса.

По #3 — исходное предложение учащегося «новый формат документа +
spaced recall механика» проблематизировано через четыре кандидатных
диагноза:
- (1) knowledge без surface (нет артефакта-носителя);
- (2) темп генерации знания превышает темп переваривания;
- (3) делегация Claude Code снимает мышечное усилие письма;
- (4) recognition без recall.
Решения для этих диагнозов разные и частично противоречат друг другу
(например, для (2) нужно сократить knowledge-генерацию, не добавить
документ).

Самостоятельный методологический ход учащегося в роли соавтора: после
проблематизации **сузил собственное предложение** от «новый формат +
механика recall» до «краткий список в конце для сверки», явно опознав
встречный риск (соответствует диагнозу 2 — «знание ожидаемого артефакта
может негативно влиять на волю к мыследеятельности»).

Сужение оставило неразобранными два вопроса: что именно записывается
в список (факт / различение / маркер открытого направления — три
разных формата), и какую функцию список выполняет (readiness-check
параллельно heat-map / backlog T-проверки параллельно ledger /
anti-amnesia — третья, не существующая в текущей методологии). Учащийся
не выбрал между интерпретациями, что переводит вопрос в эмпирический
эксперимент, не спекулятивный.

По #4 — точная конкретизация открытого вопроса 2026-05-01.

### Текущие диагнозы / гипотезы

**Гипотеза 1 (knowledge-digest):** существует функция, не покрытая
текущими артефактами курса, — anti-amnesia (запись, восстанавливающая
у учащегося ощущение прочности владения материалом, генерируемым в
сессии). Heat-map покрывает «что освоено». Evidence-ledger покрывает
«claims с уровнем R/T/C». Reflection-log покрывает «сдвиги».
Простой список знаниевых пунктов сессии — не покрывается ни одним.
Неясно: достаточная это причина для нового артефакт-формата в ядре,
или это симптом другого диагноза (см. ниже).

**Гипотеза 2 (диагноз 2 как корневой):** ощущение «поверхностно» —
не отсутствие surface, а превышение темпа генерации знания над темпом
переваривания. M1.9 содержала 4–5 значительных knowledge-блоков
([Наставник] подрежим WireMock + [Критик] о XPath/templating + два
[Симулянт-эксперт] о теоретическом потолке и SKIP LOCKED + [Performance
Engineer] о Claim Scope) в 110 минут. Решение для этого диагноза —
не новый артефакт, а пересмотр плотности knowledge-генерации в одной
сессии.

**Гипотеза 3 (диагноз 3 как корневой):** делегация Claude Code в
in-scope-частях (JSON-маппинги по форме, фактчек логов) снимает
мышечное усилие, на котором держится прочность владения. Решение —
изменение границы делегации, не введение нового документа.

Эти три гипотезы конкурируют. Дифференциация — через эмпирический
эксперимент M1.10–M1.12.

## [2026-05-03] — Macro-структура модуля как ответ на M1.7-Q2: синтез FPF и литературы по вопрошанию. Открытая гипотеза.

### Контекст

Исследовательский диалог в проекте Methodology Development. Соавтор
методологии принёс три тематически связанных пожелания:
(1) различение слоя знаний/умений и слоя онтологии в работе курса;
(2) накопление компетенций между курсами;
(3) усиление педагогической стороны методологии (Hattie, активное

В ходе диагностики обнаружено, что (3) уже идёт линией работы (записи

содержанием диалога стала (1) в связке с пометкой про 80/20.

### Связь с M1.7-Q2

При просмотре сессии M1.7 (Load Testing, 2026-05-02) обнаружено
прямое пересечение: открытый вопрос M1.7-Q2 — "может ли курс

механику, а не сопротивляться ей?" — формулирует ту же задачу,

Учащийся подтвердил, что сегодняшний запрос является содержательным
ответом на M1.7-Q2 (вариант (I) в развилке методолога), а не
рационализацией паттерна обходных ходов.

### Ход обсуждения

Первичная реформулировка тезиса 80/20 → 20/80 предложена методологом
как "смещение центра тяжести между позициями Claude от [Наставник]
к [Критик]/[Оппонент] при сохранении всех режимов". Учащийся
запросил проверку через теоретические основания (FPF и литература
по вопрошанию Карастелёва-Даниловой) — этот запрос корректен и
снимает риск принятия "правдоподобно звучащей" формулировки без
теоретической опоры.

После сверки реформулировка признана **частично верной, но слишком
грубой**.

### Ключевые находки из теоретических оснований

**FPF (FPF-Spec.md):**

- Intellect Stack (5 слоёв: Structure / Knowledge / Action / Strategy
  / Governance) — **pedagogical, not prescriptive**: вход в любой
  слой, мастерство растёт через **взаимное усиление**, не через
  линейную последовательность фаз.
- Didactic Architecture (E.6): On-Ramp first → Kernel → Extension
  Patterns → Authoring last. Это **градиент** от неформального к
  формальному, не бинарная фаза. Critically: "document order is not
  the same thing as first-practical entry".
- Tell-Show-Show (E.7): каждый паттерн **обязан** иметь structure
  universal rule → System example → Episteme example. На уровне
  атома объяснения — **T → R → R'**, не "сначала R, потом T".
- Physical grounding: "Without that anchor, the skill is rhetoric,
  not capability". Прямое подтверждение интуиции учащегося:
  матчасть — основа способности, не её антипод.

**Карастелёв-Данилова (Навигатор + Практикум):**

- Этапы развития вопрошания в ОДИ: вопросы на понимание + критика
  в форме вопросов → дифференциация по доскам (оргдеятельностной /
  объектно-онтологической) → превращение вопросов в проектные
  задачи. Это **разворачивание типов вопросов**, не разделение на
  фазы знания vs онтологии.
- Вопросное восхождение (Практикум): начальные → группирующие →
  итоговые. Динамика **внутри** работы с темой.
- QFT-сценарий Стохофа: активизация существующих знаний →
  построение своей интеллект-карты → постановка вопросов →
  поиск ответов → сравнение начальной и финальной карт. Прямо
  видна последовательность R-опора → T-проблематизация →
  онтологический сдвиг через сравнение.
- Этапная динамика темы (Навигатор): "Любое исследование проходит
  стадию формирования конкретного содержания темы... этапное
  движение от общей темы к определённой проблеме и
  специализированным задачам".

### Уточнённая формулировка

Линейность 80/20 → 20/80 как разделение фаз — теорией **опровергается**.
Ни FPF, ни Карастелёв не структурируют освоение как "сначала R,
потом T".

Динамика **опоры на материал** — теорией **поддерживается**. На
каждом масштабе работы нужна R-опора — material grounding (FPF),
активизация существующих знаний (QFT), вход в живое содержание
(ОДИ). Без неё мысле-работа становится "риторикой, не
способностью" (FPF) и не может развернуться (Карастелёв).

**Принципиальное уточнение — три масштаба R-опоры:**

- **Micro** (внутри объяснения) — Tell-Show-Show из FPF: краткое
  универсальное правило → System example → Episteme example.
  Подрежим прямого ввода материала (B-правка от 2026-05-01) — это
  уже фрагмент этого слоя, но без обязательной "Tell"-рамки до
  "Show".
- **Medium** (внутри сессии) — установка/содержание/рефлексия из
  Карастелёва-Даниловой. У нас СМД-цикл из 6 этапов. Соотношение
  с рамкой Карастелёва не разбиралось.
- **Macro** (внутри модуля) — активизация → проблематизация →
  онтологизация → закрепление. **Этого слоя в текущем ядре нет.**
  Heat-map (4 цвета) — это **диагностика**, не **последовательность
  работы**. Программа модуля задаётся темами и priority, не
  структурой работы со знанием.

### Гипотеза для будущей проверки (потенциальная D-правка)

> Курс не противопоставляет матчасть и онтологическую работу.
> Он строит R-опору на каждом масштабе — от объяснения до модуля.
> Потребность учащегося в "показуемом результате" — это потребность
> в R-видимости. Когда R-опора встроена, эта потребность работает
> как двигатель, а не как сопротивление методологии.

### Переосмысление трёх идущих линий

В новой рамке три уже идущие линии работы становятся **фрагментами
одного ответа** на M1.7-Q2, а не независимыми улучшениями по Hattie:

1. **Подрежим прямого ввода материала** (B-пилот, M1.6) — R-опора на
   micro-уровне. Кандидат на расширение: добавить Tell-перед-Show.
2. **Success Criteria в Этапе 1** (план) — критерий, что R-опора
   состоялась, а не формальный чек.
3. **SOLO как ось диагностики** (план) — структурность освоения на
   macro-уровне, вторая ось рядом с heat-map (heat-map = что
   освоено, SOLO = насколько структурно).

К этим трём добавляется **четвёртый кандидат**, которого не было в
плане:

4. **Macro-структура модуля** — последовательность активизация →
   проблематизация → онтологизация → закрепление. Потенциальная
   D-правка `core/methodology-method.md` или `core/course-template.md`.

### Что НЕ принято решением

- Гипотеза macro-структуры модуля не оформляется как правка ядра

### Условия эмпирической проверки гипотезы

## [2026-04-26] — Микро-цикл рефлексии [Наставника] на пересечении ММК и Hattie. Открытый вопрос.

### Контекст

Пользователь обозначил дефицит в работе [Наставника] на уровне отдельного объяснения: отсутствие калибровки сложности под текущий уровень, непрояснённость критериев успеха объяснения, отсутствие проверки усвоения. Предложил рассмотреть Hattie's Visible Learning как источник инструментов. По ходу диалога переформулирована задача и проверено пересечение с аппаратом ММК.

### Ход обсуждения

**Стартовая постановка** — "переработать инструкции, добавив методику видимого обучения Hattie". Проблематизирована: уровень коррекции D без эмпирической основы; неясный статус Hattie относительно СМД и FPF; контекстная асимметрия (Hattie — школьный массовый класс, у нас индивидуальное взрослое обучение).

**Уточнение рамки** — Hattie зафиксирован как инструментальный слой внутри существующих позиций (преимущественно [Наставник]), не как третья теоретическая рамка. Bounded Context: Hattie работает *внутри* СМД и FPF, не *рядом* с ними. При конфликте приоритет у фундаментальных рамок.

**Содержательный синтез по Hattie** — выделена триада формативной оценки (Where am I going / How am I going / Where to next) и четыре уровня feedback (Task / Process / Self-regulation / Self), принцип "+1 above" для калибровки, принцип disconfirmation > confirmation, требование проверки получения feedback. Зафиксировано три напряжения с существующей методологией:
- praise vs encouragement (адаптация: переносится дисциплина "не смешивать в одной фразе", не переносится тезис о вреде поддержки как таковой);
- "teacher as activator" vs СМД-позиционность (Hattie прав про explicit instruction на surface уровне, но deep / conceptual работа остаётся через мыследеятельность);
- единица измерения — урок vs сессия (микро-цикл Hattie применяется к отдельному объяснению внутри сессии).

**Проверка через ММК** — обнаружен пробел в проектном дистилляте `theoretical-foundations.md`: общий аппарат рефлексии Щедровицкого описан, но рефлексия педагога специально не разработана. Разведка дала ключевой источник — Н.Г. Алексеев (один из основателей ММК, друг Щ., докторская диссертация 2002 г. "Проектирование условий развития рефлексивного мышления"), специально занимавшийся рефлексивным подходом в образовании.

**Расширение дистиллята** — добавлена минимальная секция "Рефлексия педагога — частный случай рефлексивного выхода" в `theoretical-foundations.md`, фиксирующая онтологию (Щедровицкий + ключевые сдвиги по Локку/Канту/Фихте/Гегелю по Алексееву). **Без техник** — техники Алексеева требуют отдельного чтения первоисточников.

### Гипотеза для проверки

Микро-процедура рефлексивного контроля качества объяснения [Наставника] устроена как **рефлексивный выход [Наставника] в позицию [Методолога] относительно собственного объяснения**, на масштабе отдельного смыслового узла внутри сессии.

- триггеры рефлексивного выхода (после прохождения смыслового узла; в духе Канта — на сбое автоматизма)

## [YYYY-MM-DD] — Реструктуризация: выделение course-agnostic ядра

### Контекст

В ходе формирования методологии обнаружена архитектурная ошибка: методологический проект ("Methodology Development") был привязан к конкретному первому курсу (Java + Spring Boot 3). Инструкции проекта содержали программу конкретного курса, позиционную карту Java-мира, специфические модули (G/X/M) и т.п.

Ошибка вскрыта при попытке начать первый методологический диалог в соответствующем Claude Project: Claude справедливо указал, что работа без эмпирики (курс ещё не начался, reflection-log пуст) спекулятивна, и что привязка методологии к одному курсу нарушает изначальную заявку "шаблон для создания шаблонов".

### Ход обсуждения

Проблема осознана через честное наблюдение Claude из методологического проекта. Пользователь подтвердил фундаментальность ошибки. Принято решение провести структурную ревизию **до** начала первого курса — чтобы не накопить технический долг, исправление которого потом будет дороже.

Рассматривались два варианта:
- **A**: подождать с мета-работой до окончания Модуля 0 первого курса (дисциплина "без эмпирики не проектируем")
- **A'**: сделать только структурную ревизию сейчас, без новых спекулятивных конструкций — исправить онтологическую ошибку, не добавляя непроверенного

Выбран вариант A'. Обоснование: сама ревизия не требует эмпирики (это исправление уже вскрытой архитектурной проблемы), но блокирует развитие, если её не сделать.

### Принятые решения (уровень D)

**Выделение методологического ядра `core/` из курсового слоя.**

Созданы документы ядра (все — course-agnostic):
- `core/theoretical-foundations.md` (перенесён из `courses/java-spring-2026/`)
- `core/methodology-method.md` — принципы методологии (новый)
- `core/session-patterns.md` — паттерны сессий (новый)

- `core/infrastructure-pattern.md` — двухрепо-архитектура как паттерн (новый)
- `core/course-template.md` — шаблон развёртывания курса (новый)

Курсовой слой (`courses/java-spring-2026/`) остаётся как инстанциация, содержит только course-specific материалы — программу, модули, позиции Java-мира, sample code и т.д.

`methodology-journal.md` переносится в корень `smd-learning/`, не внутрь курса.

**Онтологическое обоснование (FPF):**

- `core/*` = U.MethodDescription (описание Method)
- `courses/[курс]/*` = Work (выполнение Method в конкретном Bounded Context)
- Разделение соответствует Strict Distinction (A.7) и принципу Single Source of Truth

**Обоснование (СМД):**

- Методология = работа над средствами (не предметная деятельность)
- Предметные курсы = работа со средствами применительно к предмету
- Методология **объемлет** предметную работу (в терминах ММК), не сливается с ней

### Эмпирическая основа

- Наблюдение из первого методологического диалога: Claude вскрыл архитектурную проблему до того, как она закрепилась в эмпирике
- Указание пользователя на нарушение заявленного свойства методологии ("максимально независимая")
- Теоретическое основание: FPF A.1.1 (Bounded Context) прямо запрещает смешение контекстов разного уровня абстракции

Данные не из проведённого курса (курс ещё не начался), но из проверяемого наблюдения архитектурной ошибки.
